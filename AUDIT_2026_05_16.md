## 1. SCALABILITY → Scope Filter / Multi-tenant Isolation

### 1.1 Current State
Defense-in-depth via app-level middleware:
- `resolveDataScope(req)` → `{ assignedClientIds, assignedProductIds, restricted }`
- `validateTaskRecordAccess`, `validateCaseAccess`, `validateCaseProductAccess`, `validateAssignmentTargetScope`
- `activeScope` intersection (Demo Mode locks) — P11.A / P13 / P14.M series
- `getScopedOperationalUserIds()` for MANAGER/TEAM_LEADER hierarchy
- DB-level RLS deliberately **deferred** (see `project_rls_deferred_2026_04_30.md` — 12 cross-tenant code paths would break; revival requires non-superuser DB role + ≥60% test coverage + staging env).

### 1.2 Strengths
- 55/55 audited routes scoped correctly. Zero leaks.
- Consistent primitives — no controller invents its own filter.
- Recent NM-1/NM-2/P14.M series confirm the team treats scope violations as critical-severity.
- WebSocket rooms scope per `req.user.id`; cross-tenant push not observed.

### 1.3 Weaknesses / Risks
- **No DB-level second lock** (RLS deferred). A single new controller written without the middleware = full cross-tenant leak. There is no defense-in-depth at the DB tier.
- **No automated regression test** that exercises "user A from client X cannot read row owned by client Y" across all routes. Any future audit re-runs by hand.

### 1.4 Impact
A future regression (intern PR, refactor, copy-paste of a non-scoped controller) could leak cross-client data and there is no DB-side safety net. Probability low (consistent patterns) but blast radius = full tenant breach.

### 1.5 Recommendations
| Priority | Item |
|---|---|
| Long-term | Revive RLS effort. Prereqs: drop SUPERUSER on `acs_user` role, establish staging env, ≥60% test coverage, rewrite 12 cross-tenant paths with explicit `SET LOCAL acs.bypass_rls=true` plumbing. |
| Short-term | Add a `scope-isolation.test.ts` integration suite: for every authenticated route, hit it with a user whose scope excludes the test row → assert 403/404. Auto-discover routes from the Express router. |
| Now | Add a CI lint rule: any new file under `src/controllers/` must import at least one of `resolveDataScope` / `validateTaskRecordAccess` / `validateCaseAccess` — fail PR otherwise. Crude but catches accidents. |

## 2. RELIABILITY → Concurrency & Race Conditions

### 2.1 Current State
- `withTransaction` wrapper is the dominant pattern.
- `caseStatusSyncService.recalculateCaseStatus` is THE single authority on `cases.status`.
- `completeTaskOrThrowIfRevoked` helper unifies 9 mobile submit paths.
- Mobile revoke uses `FOR UPDATE` + idempotent 200 sentinel.
- `mobileSyncController` IN_PROGRESS update + recalc wrapped in single tx.
- `EnterpriseCache.caseUpdate` / `assignmentUpdate` patterns include both `case:*` AND `cases:*`.
- BullMQ workers exist (`auditLogQueue`, `reverseGeocodeQueue`).

### 2.2 Strengths
- Recent revoke/revisit hardening eliminated 90% of obvious races.
- `commission_calculations` uses `ON CONFLICT DO NOTHING` for idempotency.
- `task_assignment_history` writes are tx-wrapped.
- Mobile remote-revoke triggers `ProjectionUpdater.scheduleTaskRebuild` post-wipe.

### 2.3 Weaknesses / Risks

#### R-CRIT-1 — Phantom read in case status recalc 🔴
- **File:** [caseStatusSyncService.ts:33-118](CRM-BACKEND/src/services/caseStatusSyncService.ts#L33-L118)
- **Race:** Thread A reads child tasks at line 42 (no lock). Thread B inserts a new REVISIT task or reassigns. Thread A's formula counts the old N tasks and writes `COMPLETED` when reality is `PENDING`.
- **Probability:** **High at 50+ concurrent agents.** Bulk-assign loops, mobile sync, web admin tabs all call recalc — same case targeted by multiple paths is realistic.
- **Impact:** Case status drift; billing automation may mark a case done while open tasks exist; reverse audit trail breaks.

#### R-HIGH-1 — Invoice number atomicity unverified 🟠
- **File:** [invoicesController.ts:737](CRM-BACKEND/src/controllers/invoicesController.ts) — call to `next_invoice_number()` PL/pgSQL function.
- **Risk:** If the function doesn't use `SELECT … FOR UPDATE` on the `invoice_sequences` row, two concurrent POST /invoices issue the same number → GST audit corruption.
- **Action:** Inspect the PL/pgSQL body in `acs_db_final_version.sql`. If unlocked, wrap call site in `pg_advisory_xact_lock('invoice_seq'::regclass::oid::bigint)`.

#### R-HIGH-2 — Bulk-assign cache invalidation pre-handler 🟠
- **File:** [verificationTasks.ts:365](CRM-BACKEND/src/routes/verificationTasks.ts#L365)
- **Verified pattern:** `EnterpriseCache.invalidate(..., { synchronous: true })` is positioned as Express middleware **before** the async handler. Therefore invalidation runs **before** the route's `withTransaction` commits. A concurrent dashboard GET arriving in the window between invalidation and commit repopulates the cache from a pre-commit snapshot → stale-cache window can persist for the full TTL.
- **Fix:** Move invalidation **inside** the handler, **after** `COMMIT`:
  ```ts
  // remove middleware-position invalidation
  await client.query('COMMIT');
  await EnterpriseCache.invalidate(CacheInvalidationPatterns.assignmentUpdate, { synchronous: true });
  res.json(...);
  ```
  Same pattern should be checked on other `EnterpriseCache.invalidate(...)` middleware positions.

#### R-MED-1 — Recalc lacks isolation declaration 🟡
- Caller transactions use default READ COMMITTED. Even if locking the case row (R-CRIT-1 fix), reads of child rows can still see phantom inserts. Either SERIALIZABLE + retry or `SELECT … FROM verification_tasks WHERE case_id=$1 FOR SHARE` (block concurrent task inserts during recalc).

#### R-MED-2 — `reverseGeocodeQueue` worker race 🟡
- Concurrency=5; two workers can process the same attachment ID. Idempotent WHERE clause prevents corruption but wastes Google API calls. Acceptable but add `BullMQ jobId: 'attach_<id>'` for dedup (memory says this exists for upload-time enqueue — verify worker also respects it on retries).

#### R-LOW-1 — Cache stampede on `cases:recent:*` expiry 🟢
- 100-tab dashboard at TTL expiry → 100 cache misses → 100 DB rebuilds. Probability low; mitigation: background refresh at TTL-10min.

### 2.4 Impact
- R-CRIT-1: **silent data corruption** on `cases.status` under load.
- R-HIGH-2: **stale UI for full Redis TTL** post-bulk-assign.
- R-HIGH-1: **duplicate invoice numbers** → GST/legal exposure.

### 2.5 Recommendations
| Priority | Fix |
|---|---|
| **Critical (this week)** | R-CRIT-1: `SELECT id FROM cases WHERE id=$1 FOR UPDATE` at top of `recalculateCaseStatus`. Read child tasks after the lock. |
| **Critical (this week)** | R-HIGH-1: read `acs_db_final_version.sql` for the `next_invoice_number()` body; confirm `FOR UPDATE` lock; if absent, add advisory lock at call site. |
| **High (this sprint)** | R-HIGH-2: move all `EnterpriseCache.invalidate({synchronous:true})` from middleware position to post-COMMIT inside handlers. Audit every other route using the middleware-position pattern. |
| **Medium (next sprint)** | R-MED-1: add `FOR SHARE` on the task-status SELECT inside recalc. |
| **Medium (next sprint)** | R-MED-2: verify BullMQ `jobId` dedup on reverseGeocodeQueue retries. |
| **Low (backlog)** | R-LOW-1: background refresh on hot `cases:recent:*` keys. |

## 3. SECURITY → Session, JWT, Refresh-Token & Auth-Boundary

### 3.1 Current State
- **JWT:** HS256 (default), separate `jwtSecret` (access) vs `jwtRefreshSecret` (refresh); access TTL not pinned in audit but mobile is 24h, web likely 15min.
- **Refresh tokens:** Stored in `refresh_tokens` table with SHA256-hashed token. Rotation on use. Soft-delete via `revoked_at`.
- **Storage:** Web access in-memory + refresh in HttpOnly+SameSite cookie. Mobile both in React Native Keychain (`WHEN_UNLOCKED_THIS_DEVICE_ONLY`).
- **Login:** bcryptjs (10 rounds); Redis-based 5-attempt lockout (`login_lockout:{username}`).
- **Auth middleware:** `verifyTokenAndSetUser` checks JWT + `token_version` against DB. 5s process-local cache + Redis pub/sub for cross-worker invalidation.
- **WS:** Handshake validates JWT + tokenVersion; 10 reconnects/min/userId rate limit.
- **CSRF:** `refreshCsrfGuard` middleware on `/api/auth/refresh-token`.

### 3.2 Strengths
- Refresh-token rotation atomic (old marked `revoked_reason='rotated'` before new issued).
- Token-version bump on web logout + logout-all-mobile.
- Separate secrets for access vs refresh — refresh token can't replay into access path.
- Mobile keychain storage with device-bound flag.
- 5s auth context cache + cross-worker Redis invalidation.

### 3.3 Weaknesses / Risks

#### A-CRIT-1 — Unlimited concurrent sessions, no device binding 🔴
- **File:** `acs_db_final_version.sql` → `refresh_tokens` table schema.
- **Gap:** No `device_id` / `device_fingerprint_hash` column. No `UNIQUE(user_id, device_id) WHERE revoked_at IS NULL` partial index.
- **Result:** Unlimited rows per user. No "show my active sessions" UI. No selective device revoke.
- **Attack surface:** Stolen phone scenario — admin can only nuke ALL sessions or do nothing.

#### A-CRIT-2 — Single-device mobile logout leaves access token valid 🔴
- **File:** [mobileAuthController.ts:423-443](CRM-BACKEND/src/controllers/mobileAuthController.ts#L423-L443)
- **Verified:** When `req.body.refreshToken` is supplied, only the matching refresh row is soft-deleted. `token_version` is NOT bumped. Comment at line 438-440 explicitly acknowledges this.
- **Impact:** Stolen access token survives the user's logout for the full access-TTL window (24h on mobile). Attacker can submit fake visit records / upload false photos / call admin endpoints for 24h after the victim "logged out".
- **Fix:** Two options:
  1. **Cheap (today):** Always bump `token_version` even on single-device logout. Cost = other devices forced to re-authenticate (acceptable for security).
  2. **Right (with A-CRIT-1):** Add device binding; shorten mobile access TTL to ≤15min; rely on refresh-token revocation for the specific device.

#### A-HIGH-1 — No admin force-logout endpoint 🟠
- **Gap:** Missing `POST /api/admin/users/:id/sessions/:deviceId/revoke`.
- **Impact:** Stolen-phone incident response is slow. Bumping token_version evicts everyone; password reset is brutal.

#### A-HIGH-2 — No per-request IP/device validation 🟠
- **File:** `auth.ts` middleware does not compare `req.ip` to `refresh_tokens.ip_address`.
- **Impact:** Stolen JWT replayable from any country with zero alert.
- **Mitigation:** Mobile field agents roam (4G); strict IP-bind would break legit use. Acceptable lighter fix: **log** mismatch (`AUTH_IP_MISMATCH` audit event) for forensic trail; flag only impossible-velocity moves (Mumbai→Delhi in 5min).

#### A-HIGH-3 — Redis-only login lockout 🟠
- **File:** [authController.ts:81-134](CRM-BACKEND/src/controllers/authController.ts#L81-L134)
- **Impact:** Redis restart / flush / eviction = lockout state lost. Brute-force resumable across deploys.
- **Fix:** Mirror failed attempts to `failed_login_attempts` table (durable). Redis stays as hot-path; table as cold-storage rebuild source.

#### A-MED-1 — 5s auth-context cache window 🟡
- **File:** [auth.ts:45](CRM-BACKEND/src/middleware/auth.ts#L45) — `AUTH_CONTEXT_CACHE_TTL_MS = 5_000`.
- **Risk:** Revoked permission / deactivated user / role change can persist for up to 5s across PM2 workers if Redis pub/sub silently drops. Memory acknowledges this as acceptable.
- **Fix:** Add metric `auth_cache_invalidate_failures` + alert if pub/sub publish errors >1%.

#### A-MED-2 — Refresh-token endpoint rate limit unverified 🟡
- **File:** [auth.ts:123](CRM-BACKEND/src/routes/auth.ts) uses `refreshRateLimit` — actual limit not inspected. If high (>10/min/user), stolen refresh token can generate unlimited access tokens even when login is locked.
- **Fix:** Pin to 10/min keyed on `userId` (extracted from refresh-JWT payload), not IP.

#### A-LOW-1 — JWT algorithm not pinned explicitly 🟢
- `jwt.sign()` / `jwt.verify()` don't pass `algorithm: 'HS256'` / `algorithms: ['HS256']`. Library default works today but future bump could regress (algorithm-confusion attacks).
- **Fix:** Add explicit options to all 4 sign + 2 verify call sites. 10-line change.

### 3.4 Impact
A-CRIT-1 + A-CRIT-2 combine into a 24h post-theft exploit window with no admin remediation tool. For a field-agent CRM where phones get stolen / lost / left in cabs regularly, this is the #1 operational security gap.

### 3.5 Recommendations
| Priority | Fix |
|---|---|
| **Critical (this week)** | A-CRIT-2 cheap fix: always bump `token_version` on mobile logout. 2-line patch. |
| **High (this sprint)** | A-CRIT-1 + A-HIGH-1: design device-binding schema + admin force-logout endpoint together. |
| **High (this sprint)** | A-HIGH-3: durable `failed_login_attempts` table. |
| **High (next sprint)** | A-HIGH-2 (lite): log `AUTH_IP_MISMATCH` events even without enforcement. |
| **Medium** | A-MED-1: cache invalidation metric. A-MED-2: confirm + tighten refresh rate limit. |
| **Low** | A-LOW-1: pin JWT algorithm explicitly. |

## 8. Geo-Tagged Photo, Watermark & Reverse-Geocoding

### 8.1 Current State (per memory + agent inspection)
- Mobile capture via `react-native-vision-camera` v4 + `LocationService`. Photos baked with watermark before upload (`CameraService` + `WatermarkPreviewScreen`).
- Web `MetadataOverlay` re-renders coords + accuracy + brand strip on display; `bg-black` opaque (recent fix from `bg-black/75`).
- BullMQ `reverseGeocodeQueue` enqueues at upload-time; worker stores Google `formatted_address`.
- Redis static-map cache `gmap:{lat:6dp}:{lng:6dp}:{size}:{zoom}` TTL 7d.
- DB trigger `trg_verification_attachments_freeze_address` makes address write-once.
- `geo_location` JSONB persists altitude/speed/heading w/ NaN guards (v1.0.43).
- Auth-gated `/api/storage/:key` route (closes F-B1.1 DPDP). Mobile NEVER calls Google directly.

### 8.2 Strengths
- Google API key proven absent from mobile bundle.
- Address write-once trigger prevents post-hoc tampering of resolved addresses.
- Client SHA-256 computed pre-upload, persisted in `attachments.client_sha256`.
- Per-task photo cap (20 files).
- Location-permission denial throws `GPS_REQUIRED` — no silent fallback to "no location" upload.
- Static-map cache TTL bounded.

### 8.3 Weaknesses / Risks

#### G-HIGH-1 — GPS spoofing detection absent 🟠 (downgraded from CRIT — fraud risk real but FE has many other fraud controls)
- **File:** `crm-mobile-native/src/services/LocationService.ts` (~line 89)
- **Gap:** No `position.mocked` check (Android) / `isFromMockProvider` (iOS).
- **Attack:** FakeGPS APK → agent claims visits never made; watermark bakes false coords.
- **Fix:** Reject GPS fix where `position.mocked === true`. ~5 line change.

#### G-HIGH-2 — Server-side photo SHA-256 not computed on receive 🟠
- **File:** `CRM-BACKEND/src/controllers/verificationAttachmentController.ts:545-547` (comment references `server_sha256_hash` column)
- **Gap:** Mobile sends `client_sha256`. Backend stores it. Backend does NOT re-hash bytes on receive into `server_sha256_hash`. MITM byte-swap goes undetected.
- **Fix:** After multer writes file, compute SHA-256 of bytes, write to `server_sha256_hash`. Verify `client_sha256 == server_sha256_hash` at admin-review time; mismatch → flag for manual review.

#### G-HIGH-3 — Reverse-geocode dead-letter handling missing 🟠
- **File:** `CRM-BACKEND/src/queues/reverseGeocodeQueue.ts:146-161` — `removeOnFail: { count: 200 }`.
- **Gap:** On Google outage / key revoke, jobs fail 3× → discarded. `formatted_address` stays NULL forever (freeze trigger then prevents later resolve). No DLQ table for replay.
- **Fix:** `reverse_geocode_dlq` table + admin replay endpoint. Surface backlog count in ops dashboard.

#### G-HIGH-4 — `/api/storage/:key` lacks per-row scope re-check 🟠 (downgraded from CRIT)
- **File:** [storageController.ts:31-72](CRM-BACKEND/src/controllers/storageController.ts#L31-L72) — verified.
- **Verified:** Route checks `authenticateToken` only. Docstring acknowledges: "callers should have already passed entity-scoped auth via redirect chain." In practice the FE always routes via `/api/attachments/:id` which does scope-check, but a curl-savvy authenticated user can hit `/api/storage/:key` directly and stream any object whose key they can guess.
- **Risk:** Defense-in-depth gap. Real, but blast radius limited because storage keys are UUIDs (unguessable in practice).
- **Fix:** Add a final-mile scope check inside `streamStorageObject`: look up the owning attachment row by `storage_key`, run scope check, then stream. Or: remove the generic route entirely and force all reads through entity controllers.

#### G-MED-1 — Watermark integrity check absent post-bake 🟡 (downgraded from CRIT — overstated by agent)
- **Gap:** No `watermark_baked_at` flag on attachment row to assert "this image is watermarked." If a future code path uploads a raw photo as evidence, downstream consumers can't detect it.
- **Fix:** Add `watermark_baked_at` timestamp column; mobile sets on upload; evidence queries filter `WHERE watermark_baked_at IS NOT NULL`. (Web overlay `bg-black` is already opaque — verified clean.)

#### G-MED-2 — No accuracy threshold validation 🟡
- **File:** `WatermarkPreviewScreen.tsx` GPS ladder Tier 4 accepts accuracy=Infinity.
- **Gap:** No config rejecting accuracy > 100m / 50m for KYC.
- **Fix:** Config `maxAccuracyMeters` (default 100m); persist `accuracy_meters` on row; admin UI shows ⚠️ badge for >50m.

#### G-MED-3 — Admin location-ping FCM not signed 🟡
- **File:** `CRM-BACKEND/src/controllers/mobileLocationController.ts:61-80` (ADMIN_PING source path).
- **Gap:** Mobile trusts `requestedBy` field in FCM payload without verifying push came from backend.
- **Fix:** HMAC-sign FCM payload with shared key; mobile verifies before accepting `requestedBy` attribution.

#### G-LOW-1 — EXIF stripping has catch-and-continue fallback 🟢
- **Gap:** If piexifjs throws, upload proceeds with original EXIF. Could leak camera model / IMEI.
- **Fix:** Compare pre/post strip byte size; if no shrink → reject upload. Persist `exif_stripped` boolean for audit.

### 8.4 Impact
- G-HIGH-1: enables fraud at the source (fake visits).
- G-HIGH-2: undetectable tampering of evidence — affects legal admissibility.
- G-HIGH-3: silent address backlog during Google outage.
- G-HIGH-4: theoretical cross-client photo leak via curl (mitigated by UUID key entropy).
- G-MED-3: forgeable audit attribution on admin pings.

### 8.5 Recommendations
| Priority | Fix |
|---|---|
| **High (this sprint)** | G-HIGH-1 mock-location reject; G-HIGH-2 server-side SHA-256; G-HIGH-3 DLQ + admin replay |
| **High (next sprint)** | G-HIGH-4 final-mile scope check in `streamStorageObject` |

- **File:** [casesController.ts:665](CRM-BACKEND/src/controllers/casesController.ts#L665)
- **Pattern:** `LIMIT $N OFFSET $M` — scans M rows before returning, grows linearly with page depth.

#### P-CRIT-2 — Correlated scalar subqueries on case detail 🔴

- **Pattern:** Separate `COUNT(*)` with 3 JOINs runs sync; 10M-row `task_form_submissions` = 2-5s.

#### P-CRIT-4 — LATERAL correlated subquery on every case row 🔴
